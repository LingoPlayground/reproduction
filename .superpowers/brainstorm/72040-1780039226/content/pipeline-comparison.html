当前 vs 新方案：核心思路对比

旧方案的问题：把台词文本匹配到画布节点（含废片），匹配不可靠

🔴 旧方案（当前）
Step 1: 视频 → ASR → 剧本 JSON
(有时序信息，但后续不用)

Step 2: 剧本 JSON → 改写 JSON

Step 3: 台词 →文本匹配→ 画布节点
⚠️ 画布节点含废片/花絮，纯文本匹配不可靠

Step 4: 匹配到的节点 → seedance生成 → ffmpeg拼接
(依赖节点匹配质量)
🟢 新方案（提议）
Step 1: 视频 → ASR → 剧本 JSON
✅ 时序信息是核心，全程使用

Step 2: 剧本 JSON → 改写 JSON

Step 3: ASR 时间戳 → 确定剪辑位置
+ 场景检测(PySceneDetect)辅助
✅ 原剧视频是基础，不依赖画布节点匹配

Step 4: 画布节点 →参考图+prompt改写→ seedance
✅ 只用于视觉参考，不是视频源

Step 5: 原剧片段 + 新生成片段 → ffmpeg拼接

关键区别

维度 旧方案 新方案
视频来源 画布节点视频 原剧完整视频
剪辑位置 由节点匹配决定 由ASR时间戳决定
画布节点用途 匹配+生成两个视频 仅参考图+prompt改写
无改写镜头 下载画布节点视频 直接截取原剧片段
废片处理 LLM猜测剔除 不依赖画布节点，天然规避

新方案优势

  • 原剧视频是权威来源，不依赖画布节点质量
  • ASR时间戳是精确的ms级数据，已存在但未使用
  • 画布节点仅用于视觉参考，废片不再影响匹配
  • 无改写镜头直接复用原剧，生成成本更低

需要解决的新问题

  • ASR时间戳确定镜头边界是否足够精确？
  • 场景检测如何与ASR时间轴融合？
  • 画布节点的参考图/prompt如何映射到改写台词？
  • 拼接时转场如何处理？
